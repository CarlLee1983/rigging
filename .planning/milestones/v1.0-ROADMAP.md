## Overview

Rigging v1 交付一個 opinionated、API-first 的 TypeScript backend Reference App,用 Harness Engineering 思維把 AuthContext 強制邊界寫進框架本身。五個 phase 依研究共識依賴順序推進:先鋪 Foundation(技術棧 + DDD 紀律 + ADR 機制)、再把 Elysia app skeleton 與 DDD 四層驗證起來、接著把 Auth Foundation(BetterAuth + 雙軌 AuthContext + Runtime Guards + CVE regression suite)作為 atomic unsplittable phase ship、然後以 Agent 元專案 dogfood 驗證 harness 可複用性、最後由 Quality Gate phase 收尾測試 / CI / docs。核心論述是:錯誤的寫法在這條軌道上根本 wire 不起來。

## Milestone Summary

**Key Decisions:**

- Runtime stack locked (exact pinned): Bun 1.3.12 + Elysia 1.4.28 + Drizzle 0.45.2 + BetterAuth 1.6.5 + postgres-js 3.4.9（非 bun:sql）
- Architecture: DDD 四層 × feature vertical slice `src/{feature}/{domain,application,infrastructure,presentation}/`
- AuthContext via Elysia `.macro({ requireAuth: { resolve } })`，單一根層掛載，scope = global
- Phase 3 atomicity proven: BetterAuth + 雙軌 AuthContext + Runtime Guards + CVE regression 四件不可拆一次 land
- Adopted-scope-per-phase commit 模式：Phase 4/5 皆以「PLAN.md files_modified 外的 hardening 擴張」為 atomic-phase commit 標準做法，SUMMARY.md 顯式記錄差異

**Issues Resolved During Milestone:**

- CVE-2025-61928 class (non-owner API Key creation) — regression test 守護
- AUTH-11 session fixation — Rigging 自己 wrap `revokeSessions` 於 `ResetPasswordUseCase`（BetterAuth 1.6.5 原生不清其他 session）
- Anti-enumeration: cross-user 統一 404 RESOURCE_NOT_FOUND（不用 403）
- API Key verify 由 prefix-match 改為 SHA-256 hash-match（共用 `rig_live_` 前綴時才不會選錯列）
- VALIDATION → 422 映射（error-handler.plugin.ts 前置分支）
- Destructive spike 由 `RIGGING_RUN_DESTRUCTIVE_SPIKE` 環境變數 gate，避免 DROP SCHEMA 炸共用 DB

**Issues Deferred:**

- **Phase 04 SECURITY audit** — `$gsd-secure-phase 04` 尚未執行（Known Gap，以 standalone workflow 補票）
- **CI 首次實跑** — GitHub Actions 需 push branch + PR 後才跑第一次（G22 checklist 第 10 項）
- **REQUIREMENTS.md traceability table** — 歸檔前僅 WEB-01..04 標 Complete，其餘在 `milestones/v1.0-REQUIREMENTS.md` 統一固化（原檔被刪，traceability 狀態不再作為 live artifact 維護）

**Technical Debt Incurred:**

- **Integration tests 不用 testcontainers** — ADR 0018 記錄 adopted deviation，使用共用 Postgres + `ensure-agent-schema.ts` bootstrapper；trade-off 是簡化 CI + 本地 dev 門檻，代價是測試不完全互相隔離。若 v1.1+ 進入多租戶，需重評。
- **Phase 5 plan-level summaries partial** — `05-01/02/03` 沒單獨 `-SUMMARY.md`，資訊以 `05-SUMMARY.md` 的 Plan Completion Matrix 涵蓋；對下一 milestone retrospective 的細粒度 per-plan 分析有輕度影響。
- **Adopted scope 模式** — Phase 4/5 的 atomic-per-phase commit 雖在 SUMMARY.md 記錄，PLAN.md 的 `files_modified` 清單仍未隨變更更新；未來若要把 plan 當 audit 基準需先搞清楚這個不同步。
