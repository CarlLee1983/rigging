# Milestone v1.3: Production Hardening

**Status:** ✅ SHIPPED 2026-04-21
**Phases:** 11-13

## Overview

讓 Rigging harness 從 demo-ready 升級為 production-deployable，補上三條生產必要基礎設施：真實 email 交付（Resend adapter）、Redis 持久化限流（跨重啟/跨 instance 共享計數器）、OpenTelemetry HTTP 追蹤（OTLP 相容）。三者皆透過環境變數零程式碼切換，測試套件在無任何真實憑證下全部通過。

## Phases

### Phase 11: Resend Email Adapter

**Goal**: A developer deploying to production can configure real email delivery by setting two environment variables, and email verification and password reset flows immediately deliver to real inboxes
**Depends on**: Nothing (IEmailPort interface already exists; this is a pure adapter swap)
**Requirements**: PROD-01
**Plans**: 2 plans

Plans:
- [x] 11-01-PLAN.md — Install resend@6.12.2 + extend ConfigSchema + create ResendEmailAdapter + wire createAuthModule with conditional selection and fail-fast guard
- [x] 11-02-PLAN.md — Unit tests for ResendEmailAdapter (mock resend package) + update config drift guard test

**Success Criteria Verified:**
1. ✅ RESEND_API_KEY + RESEND_FROM_ADDRESS → ResendEmailAdapter at startup (no code change)
2. ✅ Leaving either var unset → clear startup error naming the missing var
3. ✅ Email verification delivers to real inbox — human verified 2026-04-21
4. ✅ Password reset delivers to real inbox — human verified 2026-04-21 (`/api/auth/request-password-reset`)
5. ✅ Test suite passes without real Resend API key (ConsoleEmailAdapter default)

**Key Decisions:**
- resend@6.12.2 exact pin (no caret)
- RESEND_FROM_ADDRESS validated via custom FormatRegistry email regex (TypeBox has no built-in email format)
- fail-fast guard: exactly one var set = startup error

### Phase 12: Redis Rate Limit Store

**Goal**: A developer deploying multiple instances (or restarting a single instance) can configure Redis as the rate limit backend, ensuring rate limiting state is shared and persists across restarts
**Depends on**: Nothing (rate limit plugin already exists; this upgrades its backing store)
**Requirements**: PROD-02
**Plans**: 1 plan

Plans:
- [x] 12-01-PLAN.md — Redis dependencies + config + client factory + BetterAuth secondary storage + global Elysia rate limit + tests

**Success Criteria Verified:**
1. ✅ REDIS_URL → Redis-backed store (no code change)
2. ✅ Rate limit counters survive restarts
3. ✅ Two instances sharing REDIS_URL enforce single shared budget
4. ✅ Leaving REDIS_URL unset → in-memory fallback (existing behavior preserved)
5. ✅ Test suite passes without Redis instance

**Key Decisions:**
- Optional Redis shared between BetterAuth `secondaryStorage` and global Elysia rate limit
- `createRedisClient` redacts URLs in logs

### Phase 13: OpenTelemetry Tracing

**Goal**: Every HTTP request processed by the application automatically produces an OpenTelemetry trace span, collectable by any OTLP-compatible backend without any configuration from the application developer
**Depends on**: Nothing (new middleware layer on top of existing Elysia app)
**Requirements**: PROD-03
**Plans**: 5 plans

Plans:
- [x] 13-01-PLAN.md — ADR 0020: OTel SDK manual assembly decision record (MADR 4.0)
- [x] 13-02-PLAN.md — OTel packages install (exact pin) + ConfigSchema OTEL_EXPORTER_OTLP_ENDPOINT + otel-init.ts factory
- [x] 13-03-PLAN.md — tracing.plugin.ts + app.ts wiring + main.ts initTracing call
- [x] 13-04-PLAN.md — Unit tests (tracing.plugin.test.ts) + integration tests (app-otel-tracing.test.ts, InMemorySpanExporter)
- [x] 13-05-PLAN.md — Human verification checkpoint: local Jaeger trace visualization (PROD-03 #3)

**Success Criteria Verified:**
1. ✅ OTEL_EXPORTER_OTLP_ENDPOINT → spans exported (no code change)
2. ✅ Each span includes HTTP route, method, status code, latency
3. ✅ Jaeger UI shows rigging service with expected HTTP span attributes — human confirmed 2026-04-21
4. ✅ Error spans (4xx/5xx) marked with error status
5. ✅ Test suite passes with OTel loaded; no-op when OTEL_EXPORTER_OTLP_ENDPOINT unset

**Key Decisions (ADR 0020):**
- Manual OTel SDK assembly (not `sdk-node` all-in-one): bundle size control, no Node built-in auto-instrumentation, Bun compatibility
- `InMemorySpanExporter` singleton in test helper for deterministic multi-file test runs
