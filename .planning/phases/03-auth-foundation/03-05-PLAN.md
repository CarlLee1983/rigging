# 0013. API Key storage: prefix + hash + indexed

## Context and Problem Statement

Rigging's API Key resolver must verify a raw key against persisted storage on every request. Performance requires O(log n) lookup; security requires the raw key to be non-reversible at rest; operational forensics need substrings recognizable in leaked logs.

The BetterAuth apiKey plugin default schema stores a `start` hint field but does NOT add an index on it. Without an index, prefix lookup degrades to a sequential scan as the `apiKey` table grows — impacting both resolver latency and the D-10 timing-alignment guarantee.

## Decision Drivers

- Resolver path: `SELECT ... WHERE prefix = ? AND revoked_at IS NULL AND (expires_at IS NULL OR expires_at > now())` must run O(log n) for stable per-request latency.
- D-10 timing alignment: malformed and valid-format-wrong-hash paths BOTH execute one DB SELECT; the SELECT MUST use the index so the latency baseline is uniform.
- Log-leak forensics: a single consistent prefix (`rig_live_`) across all keys enables `grep` on logs/config dumps.
- Raw key must be unrecoverable from DB: prefix stores only 8 non-sensitive chars; full `key` column stores the sha256 hash (not the raw value).

## Considered Options

- **Option A — Prefix (text, first 8 chars, indexed) + hash (text, unique) in separate columns**
- Option B — Use BetterAuth's `start` field only (no index); full-key lookup by hash
- Option C — Keyed lookup by hash only (no prefix); hash column indexed

## Decision Outcome

Chosen option: **A — Prefix (first 8 chars, indexed) + hash (unique) in separate columns**, because:

- Prefix lookup O(log n) via `index('api_keys_prefix_idx').on(table.prefix)` — applied as a post-generation patch on the BetterAuth CLI output.
- Hash column stores sha256(raw key) — reversal impossible without the raw key, matching D-23.
- `rig_live_` prefix format (D-19) is publicly safe — observable in URLs of non-sensitive log lines without disclosing the key.
- D-10 timing alignment preserved — both rejection paths execute the same index-backed SELECT.

### Consequences

Good
- Resolver latency predictable as the table grows (DEMO-04 / Phase 5 load tests can assume).
- D-10 timing guarantee stable — every code path through `verifyApiKey` goes through the same index-backed query.
- Log-leak detection trivial: `grep -r 'rig_live_' logs/` is O(1) per file.

Bad
- One-line post-CLI patch on `api-key.schema.ts` required after every `bunx @better-auth/cli generate`. Contract test `tests/contract/drizzle-schema.contract.test.ts` locks drift.

Note
- v2 multi-environment prefix (`rig_test_` / `rig_prod_`) requires a second prefix column OR a prefix-regex index — deferred to SCAF-* / PROD-*.

## Pros and Cons of the Options

### Option A — prefix + hash (chosen)
- Good: O(log n) lookup; D-10 latency uniform; forensics-friendly prefix.
- Bad: one-line index patch after CLI generation.

### Option B — BetterAuth default (no index)
- Good: no schema patch needed.
- Bad: sequential scan as table grows; D-10 timing variance increases with row count.

### Option C — hash-only lookup
- Good: simplest schema.
- Bad: no forensics prefix; no way to distinguish valid-format-wrong-hash from malformed without extra parsing; resolver code more complex.

## References

- `.planning/research/PITFALLS.md` #4 (API Key plaintext storage best practices)
- `.planning/phases/03-auth-foundation/03-CONTEXT.md` D-21
- Google Cloud API Keys: prefix + hash pattern https://cloud.google.com/api-keys/docs
- prefix.dev: How we implemented API keys https://prefix.dev/blog/how_we_implented_api_keys

# 0014. API Key hashing: sha256 explicit

## Context and Problem Statement

BetterAuth's apiKey plugin accepts multiple hashing options (including `disableKeyHashing: true`). Rigging must pin the algorithm explicitly because:

1. An Agent debugging a failed `verifyApiKey` could "try disabling hashing" as a fix — a CVE-class configuration mistake.
2. Future BetterAuth releases may change the default; silent default-tracking creates an audit-invisible drift path.
3. The `timingSafeEqual` compare in the resolver assumes fixed-length hash outputs (sha256 = 32 bytes = 64 hex chars); algorithm changes invalidate ApiKeyHash value object length invariant.

## Decision Drivers

- Audit visibility: the hashing algorithm must be grep-able in the `apiKey` plugin config.
- Non-reversibility: sha256 is collision-resistant for this threat model (API key uniqueness, not password brute-force resistance — the raw key is a high-entropy random value; sha256 is sufficient).
- Rigidity Tier 1: "no agent-driven flip to hashing: false" — ADR-pinned to block that configuration path.

## Considered Options

- **Option A — `apiKey({ hashing: 'sha256' })` explicit literal**
- Option B — Rely on BetterAuth default (implicit)
- Option C — Use Bun.password argon2id (stronger hashing)

## Decision Outcome

Chosen option: **A — explicit `hashing: 'sha256'` in auth-instance.ts**, because:

- Audit trail: ADR + explicit config makes the decision visible in both docs and code.
- Regression test coverage: `tests/integration/auth/key-hash-storage.regression.test.ts` verifies the stored hash length matches sha256 — if BetterAuth ever changes algorithm, the test fails.
- Sufficient for threat model: API keys are high-entropy random (32 bytes = 256 bits); sha256's 256-bit output provides a non-reversible mapping; argon2's slow hashing is overkill and unnecessary for cache-line-speed resolver verification.

### Consequences

Good
- Algorithm locked; configuration drift caught by ADR supersede requirement.
- Resolver `timingSafeEqual` always compares 32-byte buffers (Pitfall #3 length-mismatch pre-check simpler).
- ApiKeyHash value object length invariant (64 hex chars) is stable.

Bad
- If BetterAuth 1.6.5 does NOT support `hashing: 'sha256'` as a literal option (option may be named differently), auth-instance.ts drops it and relies on the default — noted in Plan 03-01 executor log. ADR 0014 still ships to pin the intent; re-verify algorithm via regression test.

## Pros and Cons of the Options

### Option A (chosen) — explicit sha256
- Good: audit + regression; drift caught immediately.
- Bad: one more config option; breaks if BetterAuth deprecates the literal.

### Option B — BetterAuth default
- Good: less config.
- Bad: silent default drift; agent-driven `disableKeyHashing: true` path open.

### Option C — argon2id
- Good: stronger primitive.
- Bad: slow (~100ms) per verify; wrong primitive for cache-speed resolver; Bun.password is for password flows, not API key flows.

## References

- `.planning/research/PITFALLS.md` #4
- `.planning/phases/03-auth-foundation/03-CONTEXT.md` D-23
- NIST SP 800-107 Rev. 1: hash function use https://csrc.nist.gov/publications/detail/sp/800-107/rev-1/final

# 0015. Rate limit: memory store v1 / persistent store v2

## Context and Problem Statement

Rigging must have SOME rate-limiting in v1 to be "community usable" (PROJECT.md) — brute-force login attempts, password-reset flood, verification-email spam. BetterAuth 1.6.5 ships a built-in `rateLimit` with `storage: 'memory' | 'database'`. The memory store is zero-config but single-process; the database store requires schema + maintenance but survives restarts.

Pitfall #7 (BetterAuth GitHub #2112): `/send-verification-email` is NOT covered by the built-in rate-limit. A per-email application-layer wrapper is required regardless of store choice.

## Decision Drivers

- Ship harness core without ops dependencies (PROJECT.md Out of Scope: production observability).
- Dev DX: single-process memory store requires no setup — enough for 1 developer + 1 agent workflow.
- Upgrade path must be clear: v2 PROD-02 deferred to persistent store + per-email dashboard.
- Regression coverage: Plan 03-04 does NOT include a rate-limit regression test — operational correctness is out of Phase 3 scope.

## Considered Options

- **Option A — Memory store v1 (`storage: 'memory'`) + per-email wrapper; v2 PROD-02 swaps to persistent**
- Option B — Database store v1
- Option C — No rate-limit in v1; defer entirely to v2

## Decision Outcome

Chosen option: **A — Memory store v1 + per-email wrapper**, because:

- Memory store is zero-ops: dev starts fresh with `bun run dev` and gets rate-limit "for free".
- Per-email wrapper (D-16) lives in the application layer (`RegisterUserUseCase` + `RequestPasswordResetUseCase`) so it's portable across storage implementations.
- Window + max: `{ window: 60, max: 100 }` — low enough to catch flood patterns, high enough not to hit dev iteration during normal use.
- v2 PROD-02 migration is a config-only swap: change `storage: 'memory'` to `storage: 'database'` + add the `rate_limit` table (BetterAuth auto-migrates).

### Consequences

Good
- Dev experience: no Redis, no Postgres-side config; restart clears counters (acceptable in dev).
- Code locality: per-email wrapper in use case layer is easy to test + swap.
- Log.warn on rate-limit-hit (D-16) gives operator signal without requiring a dashboard.

Bad
- Multi-process deployment bypasses memory-store limits (each process has its own counter). Documented as v2 migration requirement before first production deploy.
- Memory growth: if millions of IPs hit the endpoint before window expires, memory grows. Bounded by BetterAuth's built-in eviction; acceptable for community-usable level.

## Pros and Cons of the Options

### Option A (chosen)
- Good: zero ops; clear v2 path; per-email wrapper isolates Pitfall #7.
- Bad: single-process limitation; observability limited to log.warn.

### Option B — Database store v1
- Good: multi-process correct; production-ready.
- Bad: extra table + Drizzle migration; ops concern creeps into Phase 3 scope.

### Option C — No rate-limit
- Good: simplest.
- Bad: violates "community usable" PROJECT.md constraint; leaves brute-force open.

## References

- `.planning/research/PITFALLS.md` #7 (BetterAuth rate-limit gap on /send-verification-email, GitHub #2112)
- `.planning/phases/03-auth-foundation/03-CONTEXT.md` D-16
- BetterAuth docs https://better-auth.com/docs/concepts/rate-limit

# 0016. Trust BetterAuth session cookie defaults + AUTH-11 session-fixation resolution

## Context and Problem Statement

BetterAuth 1.6.5 ships strong cookie defaults — HttpOnly + Secure (in production NODE_ENV) + SameSite=Lax. Two risks if Rigging overrides them explicitly:

1. Override drift: Rigging's override becomes the ceiling. If BetterAuth improves defaults later (e.g. adds a new cookie attribute), Rigging must manually update the override — audit-invisible staleness.
2. Agent misconfiguration: an Agent debugging a cookie issue could "try relaxing SameSite" without understanding the security impact.

Separately, Pitfall #6 + AUTH-11 required a spike to determine whether BetterAuth 1.6.5 `POST /reset-password` (token flow) auto-revokes other sessions for the user. Context7 confirmed the `revokeOtherSessions` option is exposed ONLY on `/change-password`, not `/reset-password`. Plan 03-01 ran a spike probe (`tests/spike/reset-password-session-purge.probe.test.ts`) to answer the runtime behavior.

## Decision Drivers

- Audit visibility: pinning "trust defaults" in an ADR creates a supersede requirement on any future override.
- Threat model: session-fixation attack requires all-other-session revocation on password change / reset.
- Test coverage: `tests/integration/auth/session-fixation.regression.test.ts` (Plan 03-04) validates the outcome regardless of internal implementation.

## Considered Options

- **Option A — Trust BetterAuth defaults + pin via ADR + AUTH-11 based on spike outcome**
- Option B — Override all cookie attributes explicitly + document expectations
- Option C — Don't trust defaults; always wrap every auth flow with Rigging-side double-checks

## Decision Outcome

Chosen option: **A — Trust BetterAuth 1.6.5 defaults; pin via ADR; AUTH-11 wrapped per spike result.**

Cookie attributes: BetterAuth 1.6.5 ships HttpOnly + Secure[prod] + SameSite=Lax. Future BetterAuth default changes require new ADR supersede.
