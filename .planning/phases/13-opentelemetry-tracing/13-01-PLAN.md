**標題：**
```markdown
# 0020. OpenTelemetry SDK: manual assembly over sdk-node all-in-one
```

**Context and Problem Statement：**
說明 Rigging 是 harness template（所有 downstream 用戶都繼承其依賴），需要加入 HTTP tracing 而不引入過多 bundle size 或不必要的 Node.js 模組 patching。

**Decision Drivers：**
- Rigging 是 harness template — bundle size 是 downstream concern，每個用戶都繼承它
- 只需 Elysia-level HTTP spans，不需 auto-instrument Node 內建模組（http、dns、net）
- Exact pinning 策略（與 ioredis、resend 一致）避免 OTel 快速迭代帶入 breaking change
- Bun 相容性：sdk-node 的 auto-instrument hook 在 Bun 上未完整測試

**Considered Options（三個，粗體標示選擇的 option）：**
- **Option A — 手動組裝 4 個輕量套件**（`sdk-trace-node` + `exporter-trace-otlp-http` + `resources` + `semantic-conventions`）
- Option B — `@opentelemetry/sdk-node` all-in-one
- Option C — `@elysiajs/opentelemetry` 官方 plugin

**Decision Outcome：**
選擇 **A — 手動組裝**，理由：
- sdk-node 約 2-3MB+，auto-patches Node http/dns/net — Rigging 用戶繼承這些 side effects
- 只需 Elysia lifecycle hooks level 的 spans，無需 Node 模組層 auto-instrumentation
- 官方 @elysiajs/opentelemetry plugin 底層也是 sdk-node（D-01 明確排除）
- 手動組裝讓版本 pinning 更精確，可針對每個套件獨立升級

**Consequences（Good/Bad/Note）：**
- Good: bundle 精簡（4 個套件 vs sdk-node 的 all-in-one）
- Good: 無 Node 內建模組 auto-patching side effects
- Good: 每個套件可獨立 exact-pin，避免 OTel 快速迭代引入 breaking change
- Bad: 手動管理 4 個套件版本（相互需要對齊 OTel 2.x 同一 major）
- Note: `sdk-trace-base@2.7.0` 是 `sdk-trace-node` 的傳遞依賴，不需明確列在 package.json
